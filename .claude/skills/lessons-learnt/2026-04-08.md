# Lessons Learnt — 2026-04-08

## Infrastructure & Deployment

**Problem:** Chatbot returned `⚠️ Claude error 404: Resource not found` on every message.
**Root cause:** The deployed container image was built before the codebase was migrated from the Anthropic SDK (`POST /models/v1/messages`) to the OpenAI SDK (`POST /models/chat/completions`). The container was calling the Anthropic-format endpoint, but `mychatbotdemio-resource` had no Claude model deployed — only grok-3 and DeepSeek-R1.
**Fix:** Rebuilt and pushed the container image to pick up the OpenAI SDK migration. Updated `AZURE_AI_MODEL` to a model that is actually deployed.
**Lesson:** When switching LLM SDKs, the container image must be rebuilt. Changing env vars alone doesn't change which HTTP path the SDK calls. Always verify the running container's SDK version matches the configured endpoint format (`/v1/messages` = Anthropic, `/chat/completions` = OpenAI).

---

**Problem:** DuckDB MCP health check timed out at startup — `DuckDB MCP unavailable: timed out`.
**Root cause (1):** `MCP_BASE` env var was `http://kg-mcp-azure-dev-app` (no port). MCP server listens on port 8001. Default HTTP port 80 had nothing listening → timeout.
**Root cause (2):** After adding `:8001`, still timed out. Azure Container Apps internal ingress listens on **port 80** and proxies to the container's `targetPort` (8001). Specifying `:8001` bypasses the ingress proxy entirely — the container port is not directly reachable between apps.
**Fix:** Reverted to `http://kg-mcp-azure-dev-app` (port 80). Container Apps ingress handles the port mapping transparently.
**Lesson:** In Azure Container Apps, internal service-to-service calls always go to port 80 (HTTP) regardless of what port the container listens on. `targetPort` in the ingress config is the container-side port, not the caller-side port. Never specify an explicit port when calling another Container App internally.

---

**Problem:** `UnsupportedToolUse` error — DeepSeek-R1 rejected every request that included tools.
**Root cause:** DeepSeek-R1 on Azure AI Foundry is deployed as an inference-only model — it does not support function/tool calling. The chatbot's agentic loop always sends tools in every LLM request.
**Fix:** Switched `AZURE_AI_MODEL` to `grok-3`, which is deployed on the same Azure AI resource and supports tool/function calling.
**Lesson:** Before choosing a model for an agentic chatbot, verify it supports tool/function calling on the specific deployment. On Azure AI Foundry, model capability support varies by deployment — the same model elsewhere may support tools but not in a particular Azure deployment.

## Security & Configuration

**Problem:** Snowflake MCP returned `HTTP 401 — Token validation failed: Invalid issuer`.
**Root cause:** The KG MCP Server app registration had `accessTokenAcceptedVersion = 1`. v1 tokens are issued with issuer `https://sts.windows.net/{tenant}/`, but the MCP server validated against the v2 issuer `https://login.microsoftonline.com/{tenant}/v2.0`. Mismatch → 401.
**Fix:** Updated the app registration via Microsoft Graph API: `PATCH /applications/{objectId}` with `{"api": {"requestedAccessTokenVersion": 2}}`.
**Lesson:** Azure AD app registrations default to `accessTokenAcceptedVersion = null` (treated as 1). This issues v1 tokens with the older `sts.windows.net` issuer. If your token validator expects the v2 issuer, set `requestedAccessTokenVersion = 2` in the manifest. The `az ad app update` CLI does not support this field — use `az rest --method PATCH` against the Graph API directly.

---

**Problem:** After fixing the issuer, MCP returned `HTTP 401 — Audience doesn't match`.
**Root cause:** With `accessTokenAcceptedVersion = 2`, the `aud` claim in the token changes. For v2 tokens, `aud` is the **appId** (GUID: `34e2c137-cb3b-43db-911f-1db9f2200b85`), not the identifier URI (`api://7cb00da4.../kg-mcp-server-azure-dev`). The MCP server validated `audience = ENTRA_MCP_API_URI` (the identifier URI) → mismatch.
**Fix:** Updated the MCP server's `ENTRA_MCP_API_URI` env var to the appId GUID. The chatbot's `ENTRA_MCP_API_URI` was left as the identifier URI (used to build the OAuth scope, which must be the URI form).
**Lesson:** In v1 tokens, `aud` = identifier URI. In v2 tokens, `aud` = appId (GUID). These are different values. If you validate the audience claim, the validator must use the appId when tokens are v2. The scope used to *request* the token (`api://identifier/.default`) and the audience *in* the token (appId) are different — don't conflate them. Use separate env vars if the same integration needs both.

---

**Problem:** Snowflake OAuth integration had the wrong redirect URI — pointed to old AWS Bedrock AgentCore endpoint.
**Root cause:** Integration `DEMO_MCP_OAUTH` was created for the original Bedrock-based architecture. When migrating to direct Snowflake MCP calls from the chatbot, a new integration (`CHATBOT_AZURE_MCP_OAUTH`) was created with the correct chatbot redirect URI, but credentials were initially pulled from the wrong integration (`CLAUDE_MCP_OAUTH`).
**Fix:** Created `CHATBOT_AZURE_MCP_OAUTH` with `OAUTH_REDIRECT_URI = https://chatbot-azure-dev-app.../auth/snowflake/callback`. Used `SYSTEM$SHOW_OAUTH_CLIENT_SECRETS('CHATBOT_AZURE_MCP_OAUTH')` to get the correct client credentials.
**Lesson:** When multiple OAuth integrations exist in Snowflake, always verify which integration's credentials you are using with `DESCRIBE INTEGRATION <name>` before updating the app. The redirect URI in the integration must exactly match the one registered in the app code — one wrong integration and the OAuth flow silently fails with a state mismatch.

## Application & Code

**Problem:** LLM responded "I don't have access to DEMO_DB" when user asked about Snowflake business data.
**Root cause (1):** Old `SnowflakeCapability` failed `startup_check()` (Azure AI Foundry gateway unreachable), so it was excluded from active capabilities. The LLM never received Snowflake tools or context.
**Root cause (2):** DuckDB schema context contains a table named `snowflake_usage` (Snowflake *cost metrics*). When the LLM saw "Snowflake" in a question, it routed to DuckDB and answered from there, confusing cost usage data with business data in DEMO_DB.
**Fix (1):** Rewrote `SnowflakeCapability` to call the Snowflake Cortex MCP server directly (SSE transport + per-user OAuth token). New `startup_check()` only verifies the env var is set — no gateway dependency.
**Fix (2):** Added disambiguation to DuckDB schema context: `snowflake_usage` comment now reads "cost/usage metrics only; NOT the same as Snowflake DEMO_DB business data (use snowflake_run_sql for that)". Also added routing rule to `system_fragment()`.
**Lesson:** When two capabilities share a keyword (here: "snowflake"), the LLM will route to whichever one it sees first in the system prompt. Make disambiguation explicit in the schema context comment and in `system_fragment()` routing rules — don't rely on the LLM to infer intent from capability names alone.

---

**Problem:** Snowflake schema hardcoded in system prompt would be wrong for different users due to table/column level access control.
**Root cause:** Initial design (like Dremio) planned to hardcode the schema catalog. But Snowflake has per-user role-based access — `DEMO_READER` for different users may see different tables and masked columns.
**Fix:** Implemented per-user schema discovery:
1. Added `dynamic_fragment(user_id: str) -> str` to `Capability` base class — called every turn, never globally cached.
2. After Snowflake OAuth callback, a background thread calls `INFORMATION_SCHEMA.COLUMNS` via the MCP server using that user's token — only visible tables/columns are returned.
3. `ChatbotCore._build_system()` appends `dynamic_fragment(user_id)` to the static system prompt cache on every turn.
**Lesson:** Static system prompt caching (`_sys_cache`) breaks when schema varies by user. Introduce a `dynamic_fragment(user_id)` escape hatch in the capability interface for anything that is per-user. `INFORMATION_SCHEMA.COLUMNS` in Snowflake naturally respects role-based access — it's the right source for per-user schema discovery without custom tooling.

## Architecture Decisions

**Decision:** No LangChain, LangGraph, orchestration frameworks, or agent SDKs — pure Python.
**Rationale:** The agentic loop is a ~40-line `while True` in `chatbot_core.py`. Framework abstractions would add more complexity than they remove. Full control over prompt construction, tool format conversion, error handling, and streaming without hidden layers breaking on version upgrades. Total dependency count: 11 packages.
**Pattern used:** Single LLM decides tool calls in sequence. Capabilities register tools into a flat list. `ChatbotCore._dispatch()` routes by tool name. Loop exits when `finish_reason == "stop"`.
**Lesson:** Reach for an agent framework only when the routing logic is genuinely complex (conditional branching, parallel sub-agents, state machines). For a linear tool-calling loop, a framework is overhead. The OpenAI tool-calling format (function definitions + `tool_calls` in responses) is sufficient structure on its own.

---

**Decision:** Snowflake Cortex MCP over SSE transport, called directly from the chatbot (no gateway intermediary).
**Rationale:** Previous architecture routed through Azure AI Foundry Agent Service gateway, which was unreliable and added a dependency. Snowflake's native MCP server supports standard SSE transport with Bearer token auth — no gateway needed.
**Implementation:** SSE connection opened in a daemon thread; JSON-RPC request POSTed to the message endpoint URL received from the first SSE event; response read from the SSE stream via a `queue.Queue`.
**Gotcha:** SSE transport requires two concurrent operations (read stream + POST message). Must use a background thread for the SSE reader; the main thread waits on a `Queue` with a timeout. Blocking on the SSE stream without a timeout will hang the WebSocket handler indefinitely.
**Lesson:** Snowflake Cortex MCP uses SSE transport (not streamable HTTP). Plan for the two-phase connection pattern at design time. Set explicit timeouts on both the SSE connection and the result queue — a hung MCP call should fail fast, not block the user's browser connection.
